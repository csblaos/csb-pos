# UAT Checklist: Cancel Approval (Self Slide + Manager Password)

เอกสารนี้ใช้ทดสอบ flow ยกเลิกออเดอร์แบบใหม่ที่มี 2 โหมด:
- `Owner/Manager` ยืนยันเองด้วย `เหตุผล + สไลด์`
- role อื่นยืนยันด้วย `อีเมล/รหัสผ่าน Manager + เหตุผล`

- Build/Commit:

## เงื่อนไขก่อนเริ่ม

1. มี user อย่างน้อย 3 บทบาท: `Owner/Manager`, `Staff` (มี `orders.update`), `Viewer` (ไม่มีสิทธิ์ยกเลิก)
2. มีออเดอร์ตัวอย่างที่อยู่ในสถานะยกเลิกได้ (`DRAFT`/`PENDING_PAYMENT`/`READY_FOR_PICKUP`/`PAID`/`PACKED`/`SHIPPED`)
3. จด `orderNo` และผล `status/paymentStatus` ก่อนและหลังทดสอบทุกเคส

---

## Case 1: Owner/Manager ยกเลิกจากหน้า `/orders/[orderId]` สำเร็จด้วยสไลด์

- Result: `PASS / FAIL`
- Order No:
- หลักฐาน:

### Steps

1. Login เป็น `Owner` หรือ `Manager` แล้วเข้า `/orders/[orderId]`
2. กด `ยกเลิกออเดอร์`
3. ใน modal กรอก `เหตุผลการยกเลิก` และลากสไลด์ยืนยันให้สุด
4. กด `ยืนยันยกเลิก`

### Expected

1. ออเดอร์ถูกเปลี่ยนเป็น `CANCELLED`
2. ถ้าสถานะก่อนหน้าเป็น `RESERVE` ต้องมี movement ปล่อยจอง (`RELEASE`) ตาม flow เดิม
3. มีข้อความสำเร็จ และหน้า refresh ข้อมูลล่าสุด

---

## Case 2: Staff ยืนยันด้วยรหัสผ่าน Manager ผิดซ้ำ ต้องไม่ยกเลิกและติด cooldown

- Result: `PASS / FAIL`
- Order No:
- หลักฐาน:

### Steps

1. Login เป็น `Staff` ที่มีสิทธิ์ส่งคำขอยกเลิก แล้วเปิด modal ยืนยันยกเลิก
2. กรอกอีเมล Manager ที่ถูกต้อง แต่รหัสผ่านผิด
3. กด `ยืนยันยกเลิก` ซ้ำจนเกินจำนวนครั้งที่ระบบกำหนด

### Expected

1. ระบบแจ้ง error ว่าอนุมัติไม่สำเร็จ
2. สถานะออเดอร์ต้องไม่เปลี่ยน
3. หลังผิดซ้ำหลายครั้ง ปุ่มยืนยันถูกพักชั่วคราวและแสดงเวลานับถอยหลังก่อนลองใหม่

---

## Case 3: Staff ใช้บัญชีผู้อนุมัติที่ไม่ใช่ Owner/Manager ต้องไม่ยกเลิก

- Result: `PASS / FAIL`
- Order No:
- หลักฐาน:

### Steps

1. เปิด modal ยืนยันยกเลิก
2. กรอกบัญชี Staff ปกติ (ไม่ใช่ Owner/Manager)
3. กด `ยืนยันยกเลิก`

### Expected

1. ระบบแจ้งว่า role ผู้อนุมัติไม่ผ่าน policy
2. สถานะออเดอร์ต้องไม่เปลี่ยน

---

## Case 4: ไม่กรอกเหตุผล ต้องไม่ส่งคำขอ (ทั้ง 2 โหมด)

- Result: `PASS / FAIL`
- Order No:
- หลักฐาน:

### Steps

1. เปิด modal ยืนยันยกเลิก
2. ทดสอบทั้งโหมดสไลด์ (Owner/Manager) และโหมดรหัสผ่าน (Staff) โดยเว้น `เหตุผลการยกเลิก`
3. กด `ยืนยันยกเลิก`

### Expected

1. ระบบไม่ส่ง request และแสดงข้อความให้กรอกข้อมูลให้ครบ
2. สถานะออเดอร์ต้องไม่เปลี่ยน

---

## Case 5: ยกเลิกจากหน้า `/orders/new` > `ออเดอร์ล่าสุด` สำเร็จตาม role

- Result: `PASS / FAIL`
- Order No:
- หลักฐาน:

### Steps

1. เข้า `/orders/new`
2. กดปุ่ม `ล่าสุด`
3. เลือกออเดอร์ที่ยังยกเลิกได้ แล้วกดปุ่ม `ยกเลิก`
4. ถ้าเป็น `Owner/Manager` ให้ทดสอบโหมดสไลด์, ถ้าเป็น `Staff` ให้ทดสอบโหมดรหัสผ่าน Manager
5. กด `ยืนยันยกเลิก`

### Expected

1. ยกเลิกสำเร็จและมีข้อความยืนยัน
2. รายการล่าสุดรีเฟรชแล้วเห็นสถานะออเดอร์อัปเดต

---

## Case 6: ตรวจสรุปอนุมัติยกเลิกในหน้า detail

- Result: `PASS / FAIL`
- Order No:
- หลักฐาน:

### Steps

1. ยกเลิกออเดอร์ให้สำเร็จ 1 รายการ (จากเคสก่อนหน้า)
2. เปิดหน้า `/orders/[orderId]` ของรายการนั้น
3. ตรวจ section `การอนุมัติยกเลิก`

### Expected

1. แสดง `เหตุผล`, `ผู้อนุมัติ`, `ผู้กดยกเลิก`, `เวลาอนุมัติ` ถูกต้อง
2. `วิธียืนยัน` ต้องตรงกับโหมดที่ใช้จริง (`สไลด์` หรือ `รหัสผ่าน Manager`)

## Case 7: ผู้ไม่มีสิทธิ์ ต้องไม่เห็นปุ่มยกเลิกใน `/orders/new` ล่าสุด

- Result: `PASS / FAIL`
- ผู้ทดสอบ:
- หลักฐาน:

### Steps

1. Login ด้วย user ที่ไม่มีสิทธิ์ `orders.update/cancel/delete`
2. เข้า `/orders/new`
3. เปิด `ล่าสุด`

### Expected

1. ในการ์ดออเดอร์ล่าสุด ต้องไม่แสดงปุ่ม `ยกเลิก`
2. user ยังใช้งานปุ่ม `เปิดสรุป` และ `ดูรายละเอียด` ได้ตามสิทธิ์เดิม
